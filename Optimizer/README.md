# Tornado Retrofit Abaqus Framework

Parametric optimization + Abaqus/CAE scripting workflow for a 19th‑century multi‑wythe masonry building with fire‑cut joists. The framework computes wind uplift demand, generates retrofit variants, runs Abaqus unit‑cell simulations (including a true no‑retrofit baseline), and produces a dashboard with safety vs. preservation tradeoffs.

## Problem Framing
Fire‑cut joists have no positive mechanical bearing under uplift; in tornado suction events the joist can slide out of the pocket and the roof can separate. This project evaluates retrofit strategies that reduce displacement without violating historic preservation constraints.

## Design Uplift Calculation
Helper function `calculate_design_uplift(span_ft, spacing_in, wind_speed_mph)` implements:

```
F_lbf = 0.6 * 0.00256 * V^2 * (Span/2) * (Spacing/12)
```

Defaults:
- Span = 40 ft
- Spacing = 16 in
- Wind = 135 mph (EF‑3)

Outputs:
- `force_lbf` and `force_n` used as applied uplift in FE models.

## Retrofit Strategies
Each candidate is a strategy with parameters that the optimizer can mutate.

- **A — Gravity Anchor**: vertical core + threaded rod, embedment depth and rod diameter.
- **B — Clamp Plate**: exterior plate + through‑bolt, plate size and bolt diameter.
- **C — Friction Pin**: diagonal adhesive pin, angle and embedment length.
- **D — Hurricane Strap**: strap tying joist to masonry, strap width/thickness/leg length.
- **E — Uplift Plate**: seat plate with screws, plate width/thickness and screw diameter.
- **F — Hybrid Rod + Holdown + Blocking**: diagonal epoxy rod plus holdown hardware and blocking.
- **G — Holdown**: direct holdown bracket/bolt at joist pocket.
- **H — Hanger + Blocking**: joist hanger package with fastener schedule and blocking length.
- **N — No‑Retrofit Baseline**: no anchor geometry; used for true baseline FE only.

Preservation impact scoring uses volume removed × a strategy multiplier (visibility penalty), with `N` set to zero impact.

## Abaqus Unit Cell Model
Geometry (meters):
- Wall: 16 in × 60 in × 13 in (W × H × T)
- Joist: 1.5 in × 9.25 in × 14 in
- Fire‑cut: modeled as physical separation (gap) so joist is free unless restrained.

Materials:
- Masonry: ρ=1900 kg/m³, E=2.0 GPa, ν=0.2
- Wood: ρ=550 kg/m³, E=10 GPa, ν=0.35
- Steel: ρ=7850 kg/m³, E=200 GPa, ν=0.3

Contact:
- General contact with friction = 0.5, hard normal contact.

Step:
- `ExplicitDynamicsStep`, time period = 0.05 s, ramped uplift.

Load:
- Vertical nodal uplift on joist top nodes (ramped with `RampAmp`).

Outputs:
- `U2` (vertical displacement)
- `S` max/min principal stresses

## Optimization & Mock Solver
`forensic_retrofit_optimizer.py` runs a genetic‑style optimization with a mock physics solver (used to build Pareto frontier and pick a winner when full Abaqus is not running). Full FE runs are done with the generated scripts.

## ROAR (Penn State) Workflow
This project is configured to run on ROAR via `srun` interactive nodes (not on the submit node).

1. Open a persistent SSH control session (MFA required):

```bash
ssh -M -S ~/.ssh/roar_mux -o ControlPersist=8h rjn5308@submit.hpc.psu.edu
```

2. Run jobs on an interactive compute node:

```bash
ssh -S ~/.ssh/roar_mux rjn5308@submit.hpc.psu.edu \
  "cd /storage/home/rjn5308/tornadoSims && \
   srun --immediate=60 -p interactive -N 1 -n 1 -c 1 --mem=8G -t 00:30:00 bash -lc 'module purge; module load abaqus/2022; abaqus cae noGUI=abaqus_run_A.py'"
```

3. Extract results:

```bash
abaqus python abaqus_extract.py -- retrofit_a retrofit_b retrofit_c abaqus_results.json
```

Notes:
- Abaqus 2024 had an `ISO-8859-1` runtime issue on ROAR; Abaqus 2022 was used for stable runs.
- Explicit MPI on ROAR caused floating point exceptions for `cpus>1`; all runs were stabilized with `cpus=1`.

## Baseline FE Run
A true no‑retrofit baseline is generated by `generate_no_retrofit_case.py` and run as:
- `no_retrofit_base_h1` with uplift applied directly to joist nodes.

Output captured in `baseline_results.json` and wired into the dashboard.

## Key Scripts
- `forensic_retrofit_optimizer.py` — optimization + Abaqus generator
- `generate_abaqus_batch.py` — best‑by‑strategy FE script generator
- `generate_no_retrofit_case.py` — baseline FE script generator
- `abaqus_hypothesis_patcher.py` — keyword‑level patcher for ties/embedded/loads
- `abaqus_postprocess.py` — exports FE snapshots (`before`, `U`, `Smax`) from an ODB
- `export_best_strategy_images.py` — runs postprocess for best case per strategy and writes `fe_images_manifest.json`
- `run_full_pipeline.sh` — one-command pipeline for optimization, Abaqus runs, extraction, image export, and dashboard build
- `run_hypothesis_sweep.sh` — A/B/C hypothesis sweep
- `run_bc_sensitivity.sh` — B/C sensitivity sweep
- `prepare_additional_sensitivity_cases.py` — generates D/E/F/G/H sensitivity scripts
- `run_additional_sensitivity.sh` — runs D/E/F/G/H sensitivity sweep
- `build_dashboard.py` — builds `retrofit_dashboard.html`

## Verification Suite
Run a quick verification pass to check joint-model sanity and detect common postprocess issues:

```bash
python3 verification_suite.py --root /Users/rebeccanapolitano/antigravityProjects/tornadoSims/codex --out verification_report.json
```

What it checks:
- Design-load formula scaling (span/spacing/wind)
- Strong-vs-weak parameter monotonicity for strategies A-H (mock solver)
- Monotonic displacement increase with increasing wind load
- Existence of safety-vs-preservation tradeoff pairs
- No-retrofit baseline failing the design load
- `abaqus_results.json` finite-value sanity
- `fe_images_manifest.json` fallback-image detection (flags when `before == U` or `before == Smax`)
